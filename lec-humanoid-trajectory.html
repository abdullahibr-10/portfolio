Trajectory Generation & Optimization

Nov 2025 MSc Mobile Robotics, UniBonn

A motion planner outputs a geometric path — a sequence of configurations. A trajectory generator converts this into a time-parameterised function q(t) that the joint controllers can follow, respecting velocity, acceleration, and jerk limits. Good trajectory design produces smooth, natural-looking motion that minimises mechanical wear and follows the dynamics of the robot.

1. Path vs Trajectory

A path is a geometric curve in configuration space C: a mapping from a parameter s ∈ [0, 1] to q(s) ∈ C. It specifies shape (waypoints, shapes of curves) but not timing. A trajectory is a path with a time parameterisation: q(t) = q(s(t)) where s(t) is a monotonically increasing function mapping time to path parameter. The trajectory determines the velocities q̇(t) and accelerations q̈(t) at each joint.

2. Polynomial Trajectories

The simplest trajectories are polynomials in time. Given boundary conditions at t=0 and t=T:

  • Cubic (3rd order): 4 boundary conditions — q(0), q̇(0), q(T), q̇(T). Continuous in position and velocity. Solve the 4×4 linear system for polynomial coefficients [a₀, a₁, a₂, a₃].
  • Quintic (5th order): 6 boundary conditions — add q̈(0) and q̈(T). Continuous in position, velocity, and acceleration. Essential when acceleration continuity matters (smooth force profiles).

Setting q̇(0) = q̇(T) = 0 and q̈(0) = q̈(T) = 0 gives a quintic trajectory starting and ending at rest — smooth and safe for slow manipulation tasks.

3. Minimum Jerk Trajectory

Minimising the integral of squared jerk (∫(d³q/dt³)² dt) subject to position boundary conditions gives the minimum jerk trajectory. Hogan (1984) showed that humans reach-to-grasp motions minimise jerk, giving "natural" smooth bell-shaped velocity profiles. The closed-form solution for q(0) = q₀, q(T) = q_f, with zero boundary velocities and accelerations:

q(t) = q₀ + (q_f − q₀) · [10τ³ − 15τ⁴ + 6τ⁵] where τ = t/T

The velocity profile is a smooth bell shape: q̇ peaks at t = T/2. Jerk is zero at t=0 and t=T. This profile is widely used in humanoid manipulation for natural, human-like arm motions.

4. Via Points and Splines

When multiple waypoints {q₀, q₁, ..., qₙ} must be passed through, the trajectory is split into segments. A natural cubic spline fits a cubic polynomial to each segment, with the constraint that position, velocity, and acceleration are continuous at each interior waypoint. The resulting system of equations is tridiagonal and solved in O(n) time.

Blending: rather than passing exactly through via points (which requires stopping), use parabolic blends (trapezoidal velocity profiles with smooth quadratic corners) or cubic blends to pass through waypoints at non-zero velocity. This trades exact waypoint passing for smooth continuous motion.

5. Time Parameterisation with Dynamic Constraints

Given a path q(s), find s(t) that maximises speed while respecting joint velocity limits |q̇ᵢ| ≤ v_max_i and acceleration limits |q̈ᵢ| ≤ a_max_i. This is the time-optimal path parameterisation (TOPP) problem. TOPP algorithms find the minimum-time traversal of a given path, producing bang-bang acceleration profiles at the velocity limits. MoveIt2's time-parameterisation step uses TOPP-RA (TOPP with Reachability Analysis), which runs in milliseconds.

6. Trajectory Optimization

Gradient-based trajectory optimisation directly minimises a cost function over the trajectory. CHOMP (Covariant Hamiltonian Optimisation for Motion Planning) uses a cost of the form:

C(q) = λ_smooth · ∫||q̈||² dt + λ_obs · ∫ c_obs(q(t)) dt

where c_obs is a penalty based on the signed distance field. Gradient descent (with preconditioning) jointly minimises smoothness and obstacle avoidance. Unlike RRT, CHOMP refines the path to be dynamically smooth and obstacle-clear — ideal for dense environments where RRT finds jerky paths. GPMP2 extends this framework with Gaussian Process motion priors.

Minimum Jerk in Practice The minimum jerk profile is implemented in the G1 lab using ROS2: generate a quintic trajectory using the 5th-order polynomial above, publish as a trajectory_msgs/JointTrajectory with correctly spaced timestamps, and pass to the robot controller. Position, velocity, and acceleration must all be specified at each waypoint for a smooth controller response.
