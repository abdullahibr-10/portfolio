Graph-Based SLAM

Jun 2026 MSc Mobile Robotics, UniBonn

SLAM — Simultaneous Localization and Mapping — is the problem of building a consistent map of an unknown environment while at the same time estimating where the robot is within it. Neither task can be done cleanly in isolation: accurate mapping requires knowing the pose, and accurate localization requires knowing the map. Three classical paradigms exist to handle this chicken-and-egg problem: Kalman filter approaches (EKF-SLAM), particle filter approaches (FastSLAM), and the graph-based least-squares approach. This lecture, based on the Robot Mapping course by Prof. Cyrill Stachniss at the University of Bonn, develops the third paradigm from scratch.

The graph-based approach has become the dominant framework in modern SLAM systems — it is the backbone of tools like g2o, GTSAM, and Ceres-based pipelines. Understanding it well means understanding how robots like autonomous cars and inspection drones actually build and correct their maps.

1. The Core Idea

The central insight of graph-based SLAM is to separate the problem into two stages: a front-end that constructs a graph from sensor data, and a back-end that optimizes the graph to find the most consistent set of robot poses. This lecture focuses entirely on the back-end.

The graph encodes the history of the robot's motion. Each node \(\mathbf{x}_i\) represents a pose of the robot at time \(t_i\) — in 2D this is \((x, y, \theta)\), in 3D it is a 6-DOF rigid body pose. Each edge between nodes \(\mathbf{x}_i\) and \(\mathbf{x}_j\) represents a spatial constraint: a measurement that tells us something about the relative pose of \(\mathbf{x}_j\) as seen from \(\mathbf{x}_i\). Edges are inherently uncertain, so each one also carries an information matrix \(\mathbf{\Lambda}_{ij}\) (the inverse covariance) that encodes how much to trust it. The goal of the back-end is then clear: find the configuration of all nodes that best satisfies all the constraints simultaneously, in the weighted least-squares sense.

2. How Edges Are Created

Two types of sensor events produce edges.

Odometry edges arise whenever the robot moves from \(\mathbf{x}_i\) to \(\mathbf{x}_{i+1}\). The wheel encoders or IMU give a noisy estimate of the relative displacement. This becomes an edge between consecutive nodes and forms the backbone of the trajectory graph.

Loop closure edges arise when the robot observes a part of the environment it has already seen from a different position. By running scan matching between the two observations, we obtain a virtual measurement of the relative pose between the two non-consecutive nodes \(\mathbf{x}_i\) and \(\mathbf{x}_j\). These long-range edges are what allow the system to correct accumulated drift over time — without them, odometry errors compound into an increasingly distorted map. Identifying which pairs of poses observe the same place is the data association problem and is one of the hardest parts of robust SLAM.

3. The Error Function

We represent poses as homogeneous transformation matrices. Let \(\mathbf{X}_i\) denote the matrix that maps a point from the frame of node \(i\) to the world frame. The relative transformation from node \(i\) to node \(j\), as predicted by the current graph, is \(\mathbf{X}_i^{-1}\mathbf{X}_j\).

The measurement on edge \((i,j)\) is \(\mathbf{Z}_{ij}\), the observed relative transformation. The edge is perfectly satisfied when the measured transform equals the predicted one: \(\mathbf{Z}_{ij} = \mathbf{X}_i^{-1}\mathbf{X}_j\). The discrepancy between measurement and prediction is the error for this edge:

\[ \mathbf{e}_{ij}(\mathbf{x}_i, \mathbf{x}_j) = \mathrm{t2v}\!\left(\mathbf{Z}_{ij}^{-1}\,(\mathbf{X}_i^{-1}\mathbf{X}_j)\right) \]

The function \(\mathrm{t2v}(\cdot)\) (transform-to-vector) converts a homogeneous matrix to the corresponding minimal vector \((\Delta x, \Delta y, \Delta\theta)\) in 2D. When the constraint is perfectly satisfied, \(\mathbf{Z}_{ij}^{-1}(\mathbf{X}_i^{-1}\mathbf{X}_j) = \mathbf{I}\) and the error is zero. Otherwise it measures how far the current poses are from satisfying this constraint.

The global objective is to minimize the sum of all weighted squared errors:

\[ \mathbf{x}^* = \arg\min_{\mathbf{x}} \sum_{(i,j)\in\mathcal{E}} \mathbf{e}_{ij}^\top\,\mathbf{\Lambda}_{ij}\,\mathbf{e}_{ij} \]

The state vector \(\mathbf{x} = (\mathbf{x}_1^\top\; \mathbf{x}_2^\top\;\cdots\;\mathbf{x}_n^\top)^\top\) stacks the poses of all \(n\) nodes. Each edge contributes to this sum weighted by its information matrix — a high-quality scan match produces a large \(\mathbf{\Lambda}_{ij}\), making that constraint dominate; a poor odometry estimate in a featureless corridor produces a small one.

4. Gauss-Newton Optimization

The error function is nonlinear in the poses, so we cannot solve it in closed form. We use the Gauss-Newton method: iteratively linearize, solve a linear system, and update the estimate. The steps are:

  1. Linearize each error function around the current estimate \(\mathbf{x}\).
  2. Construct the linear coefficient system \(\mathbf{H}\Delta\mathbf{x} = -\mathbf{b}\).
  3. Solve for the increment \(\Delta\mathbf{x}\) and update \(\mathbf{x} \leftarrow \mathbf{x} + \Delta\mathbf{x}\).
  4. Repeat until convergence.

4.1 Linearizing the Error

Taylor-expanding each error term to first order:

\[ \mathbf{e}_{ij}(\mathbf{x} + \Delta\mathbf{x}) \approx \mathbf{e}_{ij}(\mathbf{x}) + \mathbf{J}_{ij}\,\Delta\mathbf{x} \]

where \(\mathbf{J}_{ij} = \partial\mathbf{e}_{ij}/\partial\mathbf{x}\) is the Jacobian of the error with respect to the full state vector. A key structural observation: \(\mathbf{e}_{ij}\) depends only on \(\mathbf{x}_i\) and \(\mathbf{x}_j\), not on any other node's pose. Therefore \(\mathbf{J}_{ij}\) is zero everywhere except in the columns corresponding to \(\mathbf{x}_i\) and \(\mathbf{x}_j\):

\[ \mathbf{J}_{ij} = \begin{pmatrix} \mathbf{0} & \cdots & \underbrace{\dfrac{\partial\mathbf{e}}{\partial\mathbf{x}_i}}_{\mathbf{A}_{ij}} & \cdots & \underbrace{\dfrac{\partial\mathbf{e}}{\partial\mathbf{x}_j}}_{\mathbf{B}_{ij}} & \cdots & \mathbf{0} \end{pmatrix} \]

4.2 Building the Linear System

Substituting the linearized errors into the objective and minimizing over \(\Delta\mathbf{x}\) yields the normal equations \(\mathbf{H}\Delta\mathbf{x} = -\mathbf{b}\), where:

\[ \mathbf{H} = \sum_{ij} \mathbf{J}_{ij}^\top\,\mathbf{\Lambda}_{ij}\,\mathbf{J}_{ij}, \qquad \mathbf{b} = \sum_{ij} \mathbf{J}_{ij}^\top\,\mathbf{\Lambda}_{ij}\,\mathbf{e}_{ij} \]

For a single edge \((i,j)\), expanding the product \(\mathbf{J}_{ij}^\top\mathbf{\Lambda}_{ij}\mathbf{J}_{ij}\) gives a block matrix that is nonzero only in the \((i,i)\), \((j,j)\), \((i,j)\), and \((j,i)\) blocks:

\[ \mathbf{H}_{ij} = \begin{pmatrix} \mathbf{A}_{ij}^\top\mathbf{\Lambda}_{ij}\mathbf{A}_{ij} & \mathbf{A}_{ij}^\top\mathbf{\Lambda}_{ij}\mathbf{B}_{ij} \\ \mathbf{B}_{ij}^\top\mathbf{\Lambda}_{ij}\mathbf{A}_{ij} & \mathbf{B}_{ij}^\top\mathbf{\Lambda}_{ij}\mathbf{B}_{ij} \end{pmatrix} \]

Similarly, only blocks \(i\) and \(j\) of \(\mathbf{b}_{ij}\) are nonzero. This means the linear system can be built efficiently: iterate over edges, compute the four block contributions per edge, and accumulate into \(\mathbf{H}\) and \(\mathbf{b}\).

4.3 Sparsity

The accumulated matrix \(\mathbf{H}\) is sparse — its sparsity pattern mirrors the adjacency structure of the pose graph. In a typical robot trajectory, each pose is connected to only a handful of others (its odometry neighbours and a few loop closures), so \(\mathbf{H}\) has far fewer nonzero entries than a dense \(n \times n\) matrix. This enables the use of sparse linear solvers such as sparse Cholesky decomposition or conjugate gradients, making graph SLAM scalable to maps with thousands or tens of thousands of nodes.

5. Gauge Freedom and the Rank Deficiency of H

Before we can solve \(\mathbf{H}\Delta\mathbf{x} = -\mathbf{b}\), there is a fundamental problem: \(\mathbf{H}\) is not full rank. Every constraint in the graph is a relative constraint between two poses. Given any valid solution, we can rigidly translate and rotate the entire map — shifting all poses by the same global transformation — and the cost does not change. The system has no notion of where the map is in the world. This is called gauge freedom.

The fix is simple: fix one node (typically the first pose) by removing its rows and columns from \(\mathbf{H}\) and \(\mathbf{b}\). This anchors the global reference frame and makes the reduced system full rank. The choice of anchor does not affect the relative geometry of the optimized map; it only sets the absolute reference.

6. A Worked 1D Example

Nothing makes the algebra clearer than a concrete example. Consider the simplest possible graph: two nodes and one edge, all in 1D.

Setup

  • Poses: \(x_1 = 0\), \(x_2 = 0\) (initial guess)
  • Measurement: \(z_{12} = 1\) (we believe \(x_2\) is 1 unit to the right of \(x_1\))
  • Information: \(\Lambda = 2\)

In 1D the error for the single edge is simply:

\[ e_{12} = z_{12} - (x_2 - x_1) = 1 - (0 - 0) = 1 \]

The Jacobian is a row vector — the partial derivatives of \(e_{12}\) with respect to each variable:

\[ J_{12} = \begin{pmatrix} \dfrac{\partial e}{\partial x_1} & \dfrac{\partial e}{\partial x_2} \end{pmatrix} = \begin{pmatrix} 1 & -1 \end{pmatrix} \]

(Since \(e = z_{12} - x_2 + x_1\), we get \(\partial e / \partial x_1 = 1\) and \(\partial e / \partial x_2 = -1\).)

Building H and b

\[ \mathbf{b}_{12}^\top = e_{12}\,\Lambda\,J_{12} = 1 \cdot 2 \cdot \begin{pmatrix} 1 & -1 \end{pmatrix} = \begin{pmatrix} 2 & -2 \end{pmatrix} \] \[ \mathbf{H}_{12} = J_{12}^\top\,\Lambda\,J_{12} = \begin{pmatrix} 1 \\ -1 \end{pmatrix} \cdot 2 \cdot \begin{pmatrix} 1 & -1 \end{pmatrix} = \begin{pmatrix} 2 & -2 \\ -2 & 2 \end{pmatrix} \]

The Rank Problem

\(\det(\mathbf{H}) = 2 \cdot 2 - (-2)(-2) = 0\). The matrix is singular. As expected: any uniform shift of both \(x_1\) and \(x_2\) by the same amount leaves \(e_{12}\) unchanged, so there is a one-dimensional null space.

Fixing the Gauge

We fix \(x_1\) (set it as the anchor) by adding a very large "spring" to the origin for that variable. Technically we add a constraint \(\mathbf{H}^{11} += 1\), which amounts to:

\[ \mathbf{H} = \begin{pmatrix} 2 & -2 \\ -2 & 2 \end{pmatrix} + \begin{pmatrix} 1 & 0 \\ 0 & 0 \end{pmatrix} = \begin{pmatrix} 3 & -2 \\ -2 & 2 \end{pmatrix} \]

Now \(\det(\mathbf{H}) = 3 \cdot 2 - (-2)(-2) = 2 \neq 0\). The system is solvable.

Solving

We solve \(\mathbf{H}\Delta\mathbf{x} = -\mathbf{b}\), i.e.:

\[ \begin{pmatrix} 3 & -2 \\ -2 & 2 \end{pmatrix} \Delta\mathbf{x} = -\begin{pmatrix} 2 \\ -2 \end{pmatrix} = \begin{pmatrix} -2 \\ 2 \end{pmatrix} \] \[ \mathbf{H}^{-1} = \frac{1}{2}\begin{pmatrix} 2 & 2 \\ 2 & 3 \end{pmatrix} \] \[ \Delta\mathbf{x} = \frac{1}{2}\begin{pmatrix} 2 & 2 \\ 2 & 3 \end{pmatrix}\begin{pmatrix} -2 \\ 2 \end{pmatrix} = \frac{1}{2}\begin{pmatrix} 0 \\ 2 \end{pmatrix} = \begin{pmatrix} 0 \\ 1 \end{pmatrix} \]

The update is \(\mathbf{x} \leftarrow \mathbf{x} + \Delta\mathbf{x} = (0, 0) + (0, 1) = (0, 1)\).

Interpretation

The optimization moved \(x_2\) from 0 to 1, exactly satisfying the constraint \(z_{12} = 1\). The anchor \(x_1 = 0\) stayed fixed. The graph converged in a single iteration because the problem is linear — in practice, with nonlinear 2D or 3D poses, multiple iterations are needed.

Why Gauss-Newton converges quickly for SLAM
In graph SLAM the error function is "almost linear" near the solution — individual pose increments are small and the dominant nonlinearity comes from the angle wrapping in 2D rotations. Starting from an odometry-based initial guess keeps the linearization error small, and Gauss-Newton typically converges in 5–10 iterations even for large maps.

7. The Full Algorithm

optimize(x):
  while not converged:
    H, b = buildLinearSystem(x)   // accumulate edge contributions
    Δx   = solveSparse(HΔx = −b)  // sparse Cholesky or CG
    x    = x + Δx
  return x

The key to building the linear system efficiently is to iterate over edges rather than computing the full Jacobian. For each edge \((i,j)\):

  • Compute \(\mathbf{e}_{ij}\) and the local Jacobian blocks \(\mathbf{A}_{ij}\), \(\mathbf{B}_{ij}\).
  • Accumulate into \(\mathbf{b}\): block \(i\) gets \(\mathbf{A}_{ij}^\top\mathbf{\Lambda}_{ij}\mathbf{e}_{ij}\); block \(j\) gets \(\mathbf{B}_{ij}^\top\mathbf{\Lambda}_{ij}\mathbf{e}_{ij}\).
  • Accumulate into \(\mathbf{H}\): the four blocks \((i,i)\), \((i,j)\), \((j,i)\), \((j,j)\) each get their respective outer-product term.

8. Uncertainty from the Optimized Graph

After convergence, \(\mathbf{H}\) evaluated at the solution \(\mathbf{x}^*\) is the information matrix of the joint distribution over all poses. Its inverse \(\mathbf{H}^{-1}\) is the full covariance matrix. The diagonal blocks give the marginal uncertainty of each individual pose; off-diagonal blocks give the correlated uncertainties between pose pairs. In practice, computing the full inverse is expensive — sparse methods exist to recover only the required diagonal or near-diagonal blocks.

To find the relative uncertainty between two specific poses \(\mathbf{x}_i\) and \(\mathbf{x}_j\), fix \(\mathbf{x}_i\) (suppress its rows and columns from \(\mathbf{H}\)), then invert the reduced system and read off the \((j,j)\) block. This gives the covariance of \(\mathbf{x}_j\) conditioned on \(\mathbf{x}_i\) being known exactly.

9. What Makes This Work in Practice

The graph-based formulation has several properties that made it the dominant paradigm:

  • Scalability. The sparsity of \(\mathbf{H}\) means the linear system can be solved in near-linear time for planar (chain-like) graphs and scales gracefully even with many loop closures.
  • Offline and online variants. The full batch optimizer re-solves the entire graph after all data is collected. Incremental solvers (iSAM2, based on Bayes trees) update only the affected part of the factorization when new edges arrive.
  • Robustness. Outlier edges — wrong loop closures — can corrupt the entire solution. Robust kernels (Huber, Cauchy) or explicit outlier rejection (DCS, switchable constraints) are essential for real-world operation.
  • Separation of concerns. The back-end does not care whether edges came from LiDAR scan matching, visual place recognition, or GPS. Any front-end that produces relative pose measurements with associated uncertainties can feed into the same optimizer.
Reference implementation
The g2o library (Kümmerle et al., 2011) implements exactly this framework with support for 2D, 3D, and Lie-group poses, sparse Cholesky via CHOLMOD, and a plugin architecture for custom edge types. GTSAM uses factor graphs with the same underlying math and adds incremental solvers.
